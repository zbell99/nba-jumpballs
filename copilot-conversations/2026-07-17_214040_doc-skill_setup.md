**Core Functionality**:
- Scans recent code changes via git history
- Identifies new files, deletions, and structural changes in `src/`, `models/`, `data/`, `tests/`
- Compares current state against README.md
- Detects capability changes in exploratory analysis workflows

**Workflow**:
1. Identify Changes — Run git commands to detect structural modifications
2. Audit Documentation — Check README.md project structure, overview, data flow, capabilities
3. Suggest Edits — Propose specific text changes with context
4. User Review & Apply — Present suggestions for approval

**Output Format**: Structured review with sections showing current text, suggested text, and reasoning

**Constraints**:
- All changes are suggestions only; user approval required
- README is canonical source of truth
- Preserves existing code comments
- Flags breaking changes for discussion

## Design Decisions

### Semi-Automated Over Fully Automated
While the initial answer selected "Automated updates," the skill was designed as **semi-automated** (suggest edits for review) because:
- Documentation quality requires human judgment
- Prevents unintended changes to canonical README
- Aligns with AGENTS.md constraint that documentation is canonical and must be carefully maintained

### Git-Based Change Detection
The skill uses `git log` and git status to detect changes because:
- Provides clear, auditable history of what changed
- Avoids false positives from temporary code variations
- Enables "since last docs update" tracking

### README + Code Comments Focus
Scopes to README.md and inline code comments because:
- README is the canonical source (per AGENTS.md)
- Code comments document implementation details
- Focuses on most impactful documentation areas
